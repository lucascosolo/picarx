Generalized physical hypothesis testing (field_agent.py)
========================================================

BASE: origin/master (has belief-revision + self-model already merged).

ALSO IN THIS PATCH: removes the dead top-level module twins
(layer_b/field_agent.py, layer_b/audio_nodes.py, layer_b/coach.py). These
were byte-identical duplicates of the real modules in layer_b/modules/
that had to be hand-cp-synced. Nothing loads them: module_loader.py and
orchestrator.py both resolve entrypoints under layer_b/modules/ only, and
no code imports these names from the top level (shared libraries like
broker_client.py / spatial_store.py / speech_match.py are NOT twins and
stay put). Deleting the twins removes the sync hazard for good - a module
now lives in exactly one place.

WHAT CHANGED
------------
The single hard-coded ultrasonic-vs-vision probe is now one instance of a
reusable HypothesisTask framework, and a second hypothesis (veto-prone
location check) is built on the same framework to prove it generalizes.
Every probe stays bounded and strictly obeys the safety daemon.

1. HypothesisTask (minimal base class)
   - Standard lifecycle fields: type, state (init -> testing -> resolving),
     started_at, timeout.
   - run(now) is the template the agent calls each tick: enforces the
     shared timeout, then delegates to the subclass tick(now). Nothing
     runs unbounded.
   - resolve(now, resolution, why, **details) is the SINGLE exit point for
     every hypothesis type - it publishes picarx/exploration/hypothesis
     and mirrors the outcome into the decision journal, so the existing
     topic/journal contract is identical no matter how many task types
     exist.
   - _vetoed_since_start() is the shared "did the safety daemon flag us?"
     helper. Tasks only ever ask the daemon a question by moving slowly and
     watching for its vetoes; they never touch the safety socket.
   - Subclass hooks: start_choice() (journal detail), tick(), on_timeout(),
     follow_up() (what the agent does after resolution).

2. SensorDisagreementProbe - the ORIGINAL probe, migrated verbatim
   - Same 0:settle / 1:creep / 2:settle+judge state machine, same speed
     cap (PROBE_SPEED=12 <=15) and creep bound (<1s), same thresholds.
   - Same question string "ultrasonic_obstacle_vs_empty_vision", same
     d0/d1 payload, same phantom/real/inconclusive resolutions and the
     same follow-up (phantom -> cruise, otherwise -> normal evasion).
   - A safety-daemon veto during the creep still resolves it "real"
     immediately. Behaviour is unchanged; only the plumbing moved.

3. VetoProneLocationProbe - the NEW second hypothesis
   - Trigger (_maybe_start_veto_prone_probe): near path is clear but the
     spatial map says the current location's veto_count >=
     VETO_PRONE_THRESHOLD (3). Pulled read-only from spatial_store;
     fail-soft (no map / unknown place -> never triggers). Rate-limited by
     VETO_PRONE_PROBE_COOLDOWN (90s). Question: "is this area still
     blocked?"
   - Action: creep in at VETO_PROBE_SPEED = 10 (hard <=10 cap) for
     VETO_PROBE_APPROACH_SEC (1s), then STOP a safe distance early and just
     watch. The safety daemon remains the ONLY thing that physically stops
     the robot - the probe never drives through to decide "clear".
   - Resolution: a daemon veto at any point -> "still_blocked" (follow-up:
     normal evasion). No veto for the full VETO_PROBE_CLEAR_SEC (3s) watch
     window (the task timeout) -> "maybe_clear" (follow-up: resume
     cruising). on_timeout re-checks for a veto landing at the edge.

4. State machine wiring
   - The old "PROBING" state is now the single shared "HYPOTHESIS" state;
     explore_tick dispatches to _handle_hypothesis_tick, which runs the
     active task and fires its follow_up on resolution.
   - Probe-specific __init__ fields (probe_stage/until/d0/started_at) are
     gone, replaced by self.hypothesis (the active task) plus the two
     last_*_at cooldown timestamps.
   - Veto-prone trigger is checked in explore_tick right after the
     immediate-obstacle checks and before normal cruise/wander.

SAFETY
------
No path touches the safety socket. All motion is published as bounded,
vetoable intents on picarx/intent/move (speeds 12 and 10). The daemon
does every physical stop; the probes only listen for its vetoes. Motion
still never starts from LLM/fuzzy output - these triggers are pure
sensor/map state.
