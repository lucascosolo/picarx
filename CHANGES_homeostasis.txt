Homeostatic & self-preservation tools (vitals + low-power)
==========================================================

BASE: origin/master (7ed3a6d). This patch is orthogonal to the pending
bluetooth-fallback patch (it touches different code); if you apply both,
apply this on top and git am --3way will merge them. The only nearby lines
are the companion tool-topic constants / TOOLS list, which both patches
ADD to (no line is modified by both).

WHAT CHANGED
------------
Gives the LLM two tools to monitor and defend the robot's own physical
health, backed by a new health daemon and a vision curtailment hook.

1. companion.py - two new LLM tools (same mode-toggle pattern; no motion):
   * check_vital_stats -> returns a spoken summary of battery
     voltage/percent, CPU temperature, and free disk, read from the latest
     picarx/health/state (companion now subscribes and caches it). Synchronous
     read, no extra hardware access.
   * register_low_power_intent -> publishes picarx/tools/lowpower/request
     {active:true}; the robot enters low-power mode.

2. layer_b/modules/tools/health_daemon.py (new):
   * Publishes picarx/health/state every 30s: battery_v, battery_pct
     (linear SoC over 6.0-8.4V, the 2S Li-ion pack), temp_c
     (/sys/class/thermal - a cheap file read, no vcgencmd subprocess),
     disk_free_gb / disk_used_pct (shutil), and low_power.
   * Sole publisher of picarx/health/low_power. Enters low power on a
     battery HYSTERESIS (in at <=6.6V, out only above 7.0V) or a
     safety-daemon 'critical' flag, OR on the LLM's lowpower request
     (latched until the battery is healthy). Announces once on entering /
     leaving. This deterministic auto-trigger matters: self-preservation
     must not depend on the LLM being spoken to - register_low_power_intent
     is an extra proactive lever, not the only defense.

   BATTERY SOURCE (design note): it consumes the voltage already on
   picarx/state/world (the safety daemon reads the battery ADC; world_state
   republishes it). Re-reading the robot_hat ADC from a second process would
   contend with the safety daemon on I2C, so we don't by default. The direct
   ADC read you linked (SunFounder A4 divider) IS included as read_battery_adc(),
   but gated behind HEALTH_BATTERY_ADC=1 for setups that don't run world_state.

3. vision_basic.py - honors low power. Subscribes to picarx/health/low_power
   (same mutable-dict-from-callback pattern as the existing stream flag) and,
   while active, throttles the heavy SSD/YOLO pass from every 1.5s to every
   15s (the cheap Haar face pass keeps running). That's the "curtail heavy
   computer vision" behavior; any other module can honor the same topic.

4. module_registry.json: health_daemon registered under tools/.

STRICT BOUNDARY / INVARIANTS
----------------------------
- No motion from any of this; companion still emits no motion primitive.
- No new DB writers; single-writer rules untouched. health_daemon is the
  sole publisher of picarx/health/low_power.
- Battery hardware still read only by the safety daemon (no I2C contention).
- CPU budget: temp is a file read, disk is shutil, health loop is every 30s;
  and the whole point is to REDUCE CV load when power is low.
